# Vision — целевая архитектура Orange Schedule Bot

_2026-04-22. Записано со слов пользователя._

## Конечная цель

Система для руководителя стройки, которая позволяет управлять графиком работ **голосом с объекта**, без ручной правки таблиц. Клиент/подрядчики видят актуальный график по **публичной ссылке** (не файл, чтобы не было путаницы со старыми копиями).

## Главный user flow

1. Руководитель на объекте замечает: подрядчик срывает сроки
2. Голосом в Telegram-бота: «перенеси покрасочные работы на 3 дня вправо, причина — не успевают»
3. Transcription → intent → API-вызов в backend → задача обновлена, каскад по зависимостям
4. Публичная ссылка `schedule.orange.<domain>` автоматически показывает новый график
5. Клиент открывает ссылку → видит обновление

**Принцип:** никаких ручных действий в UI. Всё через голос.

## Далёкая цель: автогенерация графика из сметы

«Мы скармливаем смету от нашей компании — система автоматически:
- парсит работы из сметы
- извлекает сроки (в смете они указаны)
- создаёт задачи
- привязывает зависимости (что от чего зависит)
- формирует готовый Gantt»

Это killer feature. Не поставляется ни одним off-the-shelf инструментом, нужна кастомная разработка (парсер сметы + rule-based / LLM-классификация + инференс зависимостей).

## Визуальные требования

График должен выглядеть **как Excel-оригинал** (`source/schedule_original.xlsx`):
- Клетчатая тетрадь — каждый день = ячейка с видимой рамкой
- Дни подписаны всегда (не скрываются при уменьшении масштаба)
- Пустые клетки видны
- Плотность ~3mm/день (весь 4-месячный проект в одном экране)
- Бары окрашены по категориям
- Выходные/праздники подсвечены

**Вывод по исследованию (2026-04-22):** ни один PM-инструмент (Airtable, PlanRadar, Smartsheet, TeamGantt, GanttPRO) не даёт графический «тетрадный» стиль. Все рисуют классический Gantt — бары на белом фоне, без клеток. Чтобы получить Excel-визуал, нужен **кастомный HTML/SVG-рендер** из данных backend'а. Это обязательный компонент стека независимо от выбора CRM/PM-инструмента.

## Целевой стек (draft)

**Терминология:** backend = **n8n** (бизнес-логика). PlanRadar/Airtable = просто хранилище тикетов (как БД с удобным UI).

| Слой | Компонент | Роль |
|---|---|---|
| Вход | Telegram bot | Голос/текст от бригадира |
| Транскрибация | OpenAI Whisper (через n8n) | Голос → текст |
| NLU / intent | GPT-4 tool-use (через n8n) | Текст → JSON-интент |
| **Backend / оркестратор** | **n8n (cloud)** | Бизнес-логика, каскадные пересчёты, маршрутизация |
| Data store (тикеты) | PlanRadar _или_ Airtable | Хранит задачи, фото, статусы |
| Витрина (frontend) | Vercel + кастомный HTML | Клетчатый Gantt по публичной ссылке |
| Хостинг долгосрочно | Сайт компании (поддомен/личный кабинет) | Когда дозреет — переезд |

## Где будет жить «база задач» — открытый вопрос

**Варианты:**
- **Airtable** — текущий выбор. $0 free tier (но ограничен: 1200 записей, 2GB attachments). Нет native construction features.
- **PlanRadar** — $179/user/мес (Pro план, минимум для API-доступа). Есть Gantt, tickets, blueprint markup, mobile app для подрядчиков. Импорт из MS Project/Primavera. **Но:** Gantt View у них стандартный (бары на белом), не клетчатый — визуал всё равно рендерим сами.
- **Свой backend** (Postgres + n8n) — полный контроль, но разработка с нуля.

## Правило для рендера

Вне зависимости от выбора backend'а, **публичный график всегда = живая ссылка** на кастомный HTML-рендер, который:
- читает актуальные данные через API
- выдаёт клетчатый Gantt
- обновляется при каждой загрузке страницы

Никаких статичных файлов (`gantt.html` скачиваемый) — только URL.

## PlanRadar — слой field-issues поверх Gantt (2026-04-23)

**PlanRadar = НЕ замена Gantt, а overlay.**

### Концепция
```
HTML Gantt (schedule bars)
    + overlay layer (PlanRadar тикеты как маркеры на барах)
    + toggle on/off (галочка в UI)
```

### User flow
1. Бригадир на объекте — голос в Telegram: «проблема с электрикой — не хватает кабеля» + фото
2. n8n → Whisper+GPT → создаёт тикет в PlanRadar (нужная фаза) + прикрепляет фото
3. HTML Gantt при загрузке тянет тикеты через PlanRadar API → рисует маркеры (◆ ■) на соответствующих барах
4. Клик на маркер → popup: фото + описание + статус тикета
5. Toggle: показывать/скрывать overlay (режим «чистый план» vs «режим проблем»)

### Маркеры
- Форма: ромбик / квадратик (отличается от бара)
- Цвет: open=красный, in_progress=жёлтый, resolved=зелёный
- Позиция: на баре задачи, по дате создания тикета

### Открытый вопрос (решить при probe)
Маппинг тикет → задача в schedule.json:
- **Вариант A:** custom field в PlanRadar = `task_id` из schedule.json (лучший контрол)
- **Вариант B:** имя фазы PlanRadar = section ID в schedule.json (проще, но грубее)

### Нужно проверить в probe

- Как прикрепить фото к тикету через API (upload + attach)
- Rate limit (известно: 30 req/min)
